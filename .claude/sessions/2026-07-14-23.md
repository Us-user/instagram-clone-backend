## Цель сессии
Реализовать Phase 19 целиком (§10): административная верификация («синяя галочка») и управление
ролью Admin через отдельный `AdminController` (`[Authorize(Roles="Admin")]`). Довести до конца
инвариант #10 — отдавать `isVerified` во **всех** DTO пользователя/автора. Держать сборку зелёной,
не ломать контракт базы.

## Сделано
- **Infrastructure — сервис:** `IAdminService`/`AdminService`.
  - `VerifyUserAsync`/`UnverifyUserAsync` — переключают `User.IsVerified` через
    `UserManager.UpdateAsync`; идемпотентная защита: `BadRequest`, если уже в целевом состоянии.
  - `GrantAdminAsync`/`RevokeAdminAsync` — роль Admin через `AddToRoleAsync`/`RemoveFromRoleAsync`
    (+ `IsInRoleAsync`-проверка «уже/не админ»). На `revoke-admin` — защита от самолокаута:
    нельзя снять роль Admin с самого себя (текущий id из claims).
  - `userId` — из query; целевого юзера ищем `FindByIdAsync` → `NotFound` при отсутствии.
- **WebApi — контроллер:** `AdminController` с `[Authorize(Roles = DbInitializer.AdminRole)]` на
  классе, 4 эндпоинта дословно по контракту: `POST /Admin/verify-user?userId`,
  `DELETE /Admin/unverify-user?userId`, `POST /Admin/grant-admin?userId`,
  `DELETE /Admin/revoke-admin?userId`.
- **DI:** `services.AddScoped<IAdminService, AdminService>()`.
- **isVerified во всех author-DTO (доработка инварианта #10):** обнаружено, что `isVerified`
  отдавался только в новых DTO фаз 12–15 (`CloseFriendDto`/`GetFollowRequestDto`/`GroupMemberDto`/
  `NotificationActorDto`), а **базовые author-DTO его не отдавали**. Добавлено поле `IsVerified` и
  заполнение в:
  - `GetUserProfileDto` (профиль) — `UserProfileService.BuildProfileDtoAsync`.
  - `GetPostDto` (автор поста) — `PostProjections.ToDto` (`p.User.IsVerified`).
  - `GetPostCommentDto` (автор коммента) — `CommentProjections.ToDto`.
  - `GetStoryDto` (автор сторис) — `StoryProjections.ToDto`.
  - `GetUserDto` (поиск/подписчики/подписки/заблокированные) — `UserService`, `BlockService`,
    `FollowingRelationShipService` (2 сайта).
- **Миграция:** не потребовалась — колонка `IsVerified` в `AspNetUsers` уже существует с Phase 11
  (`AddNotifications`). Схема не менялась.
- **Сид:** `admin.IsVerified = true`, `alice.IsVerified = true` — чтобы `isVerified` был сразу
  проверяем в DTO (галочка у платформенного admin и у публичного автора alice).

## Решения
- **Роль Admin — через ASP.NET Identity, галочка — флагом на User.** verify/unverify пишут
  `User.IsVerified` (домен-флаг для отрисовки), grant/revoke оперируют ролью Identity (авторизация).
  Это два независимых понятия: верифицированный ≠ админ.
- **Идемпотентные проверки + понятные ошибки.** Как в `BlockService` (throw при «уже заблокирован»),
  для админ-действий бросаем `BadRequest` с ясным текстом, если пользователь уже в целевом состоянии
  (уже верифицирован / уже админ / не верифицирован / не админ) — понятный фидбэк админу.
- **Защита от самолокаута.** `revoke-admin` не даёт снять роль Admin с самого себя (не входит в
  контракт §10 явно, но предотвращает потерю доступа последним/текущим админом). grant-admin на себя
  и так упрётся в «уже админ».
- **AutoMapper не используется — правим проекции.** Профиль `MappingProfile` зарегистрирован в DI,
  но ни один сервис не инъектит `IMapper` (все DTO собираются вручную/через выражения-проекции для
  EF). Поэтому `isVerified` добавлялся в проекции (`*Projections.ToDto`) и ручные `new …Dto`, а не в
  `CreateMap`. `MappingProfile` не трогал (мёртвый код, конфиг не валидируется).
- **`GetStoriesDto`/`StoryItemDto` — мёртвый код.** Нигде не используются (лента сторис отдаёт
  `List<GetStoryDto>` через `StoryProjections`); не расширял.
