  - `like-post?postId` / `add-post-favorite` — тумблеры, возвращают новое состояние (`Response<bool>`).
  - `view-post?postId` — идемпотентно (уникальный индекс (PostId, UserId)); повторный вызов не дублирует.
  - `add-comment` (body `AddPostCommentDto`) — валидатор + проверка поста, возврат `GetPostCommentDto`

## Решения
- **Проекция `Post → GetPostDto` вынесена в `PostProjections.ToDto` (выражение, а не AutoMapper).**
  Нужны вычисляемые поля (счётчики, `isLiked`/`isFavorite` под конкретного юзера) — выражение
  переводится в SQL и не тянет лишние навигации. Убрало дублирование между `PostService` и
  `UserProfileService` (закрыт TODO сессии 07).
- **`get-following-post?UserId` трактуется как «лента подписок указанного UserId».** Параметр в
  контракте присутствует (как у `get-subscriptions?UserId`), поэтому показываем посты тех, на кого
  подписан переданный `UserId`; если параметр пуст — по умолчанию текущий юзер. Единообразно с
  соц-графом Phase 5 и не ломает основной сценарий «моя лента».
- **`AddPostDto.IsReel` добавлено как опциональное поле формы (default false).** Контракт `add-post`
  перечисляет Title/Content/Images; клиент, шлющий только их, получает обычный пост дословно по
  контракту. Доп-поле — единственный способ создавать reels через API (иначе только через Seed);
  чисто аддитивно, ничего не ломает.
- **`view-post` идемпотентен, а `like`/`favorite` — тумблеры.** Просмотр уникален на юзера (индекс),
  повторный вызов ничего не делает и возвращает `true`. Лайк/избранное переключаются и возвращают
  новое состояние (`true` — поставлено, `false` — снято), чтобы клиент сразу знал результат.
- **Query-параметры `int?`/`string?` + guard'ы в сервисе.** Как в Phase 4/5: non-nullable параметр
  заставил бы `[ApiController]` вернуть framework-ProblemDetails; nullable + `BadRequestException`
  прогоняет ошибку через exception-middleware в unified `Response<T>`.
- **Владелец ресурса — `ForbiddenException` (403).** `delete-post`/`delete-comment` при чужом ресурсе
  дают 403 (не 404), выполняя правило ТЗ №4 об авторизации владельца.
