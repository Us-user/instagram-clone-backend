## Цель сессии
Реализовать Phase 21 целиком (§12): content-based рекомендательный движок «как в инсте» без ML.
Профиль интересов (веса favorite>comment>like>view + затухание по времени), агрегация любимых
хэштегов/авторов, скоринг кандидатов, фильтры (свои/просмотренные/блок/приват/уже-подписан),
разнообразие, эндпоинты `GET /Explore/get-feed` и `GET /Explore/get-popular` (cold start).
Держать сборку зелёной, не заводить лишнюю миграцию.

## Сделано
- **Infrastructure — сервис `ExploreService` (`IExploreService`):**
  - `BuildInterestProfileAsync` — профиль интересов **на лету**: собирает веса постов из
    взаимодействий пользователя (favorite=4 > comment=3 > like=2 > view=1) с экспоненциальным
    затуханием по времени (полураспад 30 дней; несколько действий по посту суммируются;
    просмотры без отметки времени — без затухания), затем разворачивает вес поста в веса
    любимых хэштегов (`HashtagId`) и авторов.
  - `GetFeedAsync` — персональная лента: кандидаты = доступные посты чужих авторов, на которых
    не подписан и которые ещё не просмотрел (`VisibleTo` + `p.UserId != currentId` +
    `!followingAccepted.Contains` + `!p.Views.Any(...)`), ограничение пула `MaxCandidates=500`
    (свежайшие сверху). Скоринг `0.40·хэштеги + 0.30·автор + 0.15·популярность + 0.15·свежесть`
    (компоненты хэштеги/автор/популярность нормируются в [0..1] по максимуму пула; свежесть —
    затухание с полураспадом 7 дней). `Diversify` — не более 2 постов подряд от одного автора.
    Страница материализуется через общий `PostProjections.ToDto` + `MentionEnrichment`, порядок
    ранжирования восстанавливается по списку Id.
  - `GetPopularAsync` — cold start: чистая популярность (лайки+комменты+просмотры),
    свежесть — тай-брейк; фильтры доступа (`VisibleTo`) + исключение своих постов.
  - Хэштеги кандидатов/интересов грузятся отдельным батч-запросом `LoadHashtagIdsByPostAsync`
    (без коллекционной проекции в SQL — гарантированная трансляция Npgsql).
- **WebApi — контроллер `ExploreController`:** `GET /Explore/get-feed`, `GET /Explore/get-popular`
  (оба авторизованы по умолчанию, параметры `PageNumber`/`PageSize`).
- **DI:** `services.AddScoped<IExploreService, ExploreService>()`.
- **Сид (`DbInitializer.SeedExploreContentAsync`):** Explore-контент вокруг демо-субъекта alice —
  посты авторов вне её подписок (`carol`: закат #sunset #nature, пляж #travel #beach, город #city;
  `frank`: горы #travel #mountains) + пост подписки admin (#sunset #photography); взаимодействия
  alice (favorite+view bobReel и carolCity, like+view adminSunset) формируют профиль интересов;
  популярность (лайки/просмотры bob/admin/frank) на кандидатах. Новые хэштеги
  `photography/nature/beach/mountains/city`, базовые `sunset`/`travel` переиспользованы
  (наращён `PostsCount`).
- **Миграция:** не требуется (профиль интересов считается на лету — новых сущностей нет).

## Решения
- **Профиль интересов — на лету, без `UserInterest`.** §12 явно допускает «сущность `UserInterest`
  (агрегат) **или расчёт на лету**». Расчёт на лету исключает устаревание агрегата и лишнюю
  миграцию (как Phase 19, где миграция не понадобилась), согласуется со стилем проекта (DTO
  собираются проекциями/группировкой, а не материализованными агрегатами — как в `NotificationService`).
- **Ранжирование in-memory над ограниченным пулом.** Персональный скоринг делается в памяти над
  `MaxCandidates=500` свежайшими кандидатами — чтобы не тянуть всю таблицу; фильтры и выборка
  кандидатов транслируются в SQL. Хэштеги грузятся отдельным запросом (коллекционная проекция
  `Select(... ToList())` в одном запросе с `Take` рискованна для трансляции — избегаем).
- **Детерминированное перемешивание, без рандома.** «Перемешивание топа» реализовано затуханием
  свежести (уже в score) + диверсификацией по авторам, без случайности — иначе страницы пагинации
  перекрывались бы/теряли посты между запросами.
- **Graceful degradation для холодного юзера.** В `get-feed` у юзера без истории компоненты
  хэштегов/автора обнуляются → лента автоматически сводится к популярности+свежести. `get-popular`
  остаётся явным cold-start фолбэком (для клиента, у которого ещё нет персональной ленты).
- **`get-popular` не исключает просмотренные/подписки** (только свои/блок/приват) — это витрина
  популярного, а не «открытие нового»; исключение просмотренных/подписок — только в персональной ленте.
- **DTO не заводил** — обе выдачи возвращают `PagedResponse<List<GetPostDto>>` (как остальные ленты),
  обратная совместимость и переиспользование проекции/энричмента упоминаний.
