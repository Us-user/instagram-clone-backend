## Цель сессии
Полностью закрыть Phase 7: `StoryService` + `StoryController` со всеми 8 эндпоинтами контракта —
ленты (подписки/юзер/мои) с окном активности 24ч, создание сторис из поста или файла, удаление
автором, лайк-тумблер, уникальный просмотр, `GetStoryById` со сводкой по зрителям (`viewerDto`).
Переиспользовать `IFileService`, `ICurrentUserService`; проекцию `Story → GetStoryDto` вынести
в общий helper по образцу `PostProjections`.

## Сделано
- **`Infrastructure/Common/StoryProjections.cs`** (новый) — `ToDto()`:
  `Expression<Func<Story, GetStoryDto>>`. Автор (avatar/userName/name), агрегаты
  `viewCount`/`viewLike` в `ViewerDto`. Для сторис из поста `FileName` подставляется из первого
  изображения поста-источника (`s.FileName ?? s.Post.Images.First… ?? ""`). Переводится в SQL,
  единый источник для всех выборок сторис.
- **`IStoryService` / `StoryService`** (8 эндпоинтов):
  - `get-stories` — активные (<24ч) сторис тех, на кого подписан текущий юзер; «группировка по
    авторам» выражена сортировкой `OrderBy(UserId).ThenByDescending(CreatedAt)`.
  - `get-user-stories/{userId}` — активные сторис указанного юзера; `get-my-stories` — свои активные.
  - `LikeStory?storyId` → `Response<string>` — тумблер («Лайк добавлен.» / «Лайк убран.»).
  - `GetStoryById?id` → `Response<GetStoryDto>` (404 если нет); read-only, просмотр не фиксирует.
  - `AddStories?PostId` (multipart `Image`) — из поста (проверка существования, `PostId` set,
    `FileName` null) **или** из файла (`IFileService.SaveFileAsync`, `FileName` set, `PostId` null);
    ни того ни другого → `BadRequestException`. Возврат `GetStoryDto`.
  - `DeleteStory?id` → `Response<bool>` — только автор (`ForbiddenException`); собственный файл
    удаляется с диска после `SaveChanges` (сторис из поста своего файла не имеет — файл поста цел),
    каскады EF чистят `StoryLike`/`StoryView`.
  - `add-story-view?StoryId` → `Response<GetStoryViewDto>` — уникально на юзера (индекс
    (StoryId, ViewUserId)); повторный вызов возвращает существующую запись.
- **`StoryController`** — `[Route("[controller]")]`, дословные пути (включая PascalCase
  `AddStories`/`DeleteStory`/`LikeStory`/`GetStoryById`); `AddStories` — `[FromQuery] postId` +
  `[FromForm] AddStoryDto`; паттерн `StatusCode(result.StatusCode, result)`.
- **`Domain/DTOs/Story/AddStoryDto.cs`** (новый) — `IFormFile? Image` для multipart.
- **DI:** зарегистрирован `IStoryService → StoryService`.
- **Рантайм-смоук** (порт 5030, БД offline — старт переживает): все **8** эндпоинтов `Story` в
  `swagger.json` с дословными путями; `AddStories` — `postId` (query) + `multipart/form-data`
  (Image); `get-stories`, `GetStoryById`, `LikeStory`, `add-story-view` без токена → **401**.

## Решения
- **Проекция `Story → GetStoryDto` вынесена в `StoryProjections.ToDto` (выражение, не AutoMapper).**
  Как и `PostProjections`: вычисляемые поля (счётчики просмотров/лайков, fallback `FileName` из
  поста) переводятся в SQL. Общий источник для 5 методов сервиса, без дублирования.
- **`ViewerDto` = автор + агрегаты сторис.** `userName`/`name` — юзернейм и `FullName` автора;
  `viewCount`/`viewLike` — число просмотров/лайков сторис. Даёт самодостаточный DTO: id/avatar на
  верхнем уровне + username/name/counts внутри `viewerDto`.
- **Окно 24ч применяется во всех выборках сторис** (get-stories/get-user-stories/get-my-stories),
  т.к. по ТЗ сторис «живёт 24 часа» — просроченные исключаются единообразно. `since` захватывается
  в локальную переменную до `Where`, чтобы EF корректно параметризовал `UtcNow.AddHours(-24)`.
- **«Группировка по авторам» реализована сортировкой, а не новым DTO.** Контракт перечисляет только
  `GetStoryDto`; изобретать группирующий DTO не стал — `OrderBy(UserId).ThenByDescending(CreatedAt)`
  даёт плоский список, сгруппированный по автору и упорядоченный по свежести. Консервативно и
  дословно по контракту.
- **`AddStories`: PostId (query) ИЛИ Image (form), взаимоисключающе, приоритет у PostId.** Если задан
  валидный `PostId` — сторис из поста (файл не сохраняется); иначе если есть `Image` — из файла;
  иначе `BadRequestException`. `FileName` в проекции для post-based сторис берётся из первого
  изображения поста, чтобы клиенту было что рендерить.
- **`LikeStory` — тумблер с текстовым ответом (`Response<string>`).** Контракт задаёт `Response<string>`;
  возвращаем сообщение о новом состоянии (добавлен/убран), т.к. тип не bool.
- **`add-story-view` идемпотентен и записывает просмотр любого аутентифицированного юзера** (включая
  автора) — «уникально на юзера» по индексу; повторный вызов не дублирует и возвращает ту же запись.
- **Query/route-параметры `int?`/`string?` + guard'ы в сервисе.** Как в Phase 4–6: nullable +
  `BadRequestException` прогоняет ошибку через exception-middleware в unified `Response<T>`
  (non-nullable дал бы framework-ProblemDetails). Read-методы — `AsNoTracking()`.
