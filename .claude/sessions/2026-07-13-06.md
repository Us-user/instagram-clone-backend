(`register`, `login`, `ForgotPassword`, `ResetPassword`, `ChangePassword`) поверх готовой

## Сделано
- **`IAccountService` / `AccountService`** (`Infrastructure/Services/`):
  - `RegisterAsync` — валидация через `IValidator<RegisterDto>.ValidateAndThrowAsync`, проверка
    уникальности userName и email (`FindByNameAsync`/`FindByEmailAsync` → `BadRequestException`),
    `UserManager.CreateAsync`, `AddToRoleAsync("User")`, создание **пустого** `UserProfile`,
    выдача JWT через `ITokenService`.
  - `LoginAsync` — валидация `LoginDto`, `FindByNameAsync` + `CheckPasswordAsync` (единое
    сообщение «Неверное имя пользователя или пароль»), JWT в `data`.
  - `ForgotPasswordAsync` — `GeneratePasswordResetTokenAsync`, токен возвращается в `data` и
    пишется в лог (в учебных целях, как в ТЗ).
  - `ResetPasswordAsync` — guard'ы (token/email/password/совпадение), `ResetPasswordAsync` Identity.
  - `ChangePasswordAsync` — id текущего юзера из claims (`ICurrentUserService.GetRequiredUserId`),
    `ChangePasswordAsync` Identity.
- **`AccountController`** (`WebApi/Controllers/`): `[Route("[controller]")]`, дословные пути/методы —
  `POST register`, `POST login`, `DELETE ForgotPassword`, `DELETE ResetPassword`, `PUT ChangePassword`.
  `[AllowAnonymous]` на register/login **и** forgot/reset (см. Решения). Ответ —
  `StatusCode(result.StatusCode, result)`, чтобы честно отдавать статус из `Response<T>`.
- **DI:** зарегистрирован `IAccountService → AccountService` в `Infrastructure/DependencyInjection.cs`
  (новый блок «Сервисы фич»).
- **Рантайм-смоук** (порт 5030, БД недоступна — startup корректно это переживает):
  все 5 путей видны в `swagger.json` с верными методами; `POST /Account/login` с пустым телом →
  400 с сообщениями FluentValidation в `Response<T>`; `PUT /Account/ChangePassword` без токена → 401
  (fallback-политика); `DELETE /Account/ForgotPassword?email=` → 400 в unified-формате;
  `ResetPassword` с несовпадающими паролями → 400 «Пароли не совпадают.».

## Решения
- **Register возвращает JWT.** Контракт задаёт только вход/эффект (создать User + профиль),
  формат ответа не фиксирует; выдача токена сразу логинит клиента и единообразна с `login`.
  Это расширение входных требований не нарушает.
- **`ForgotPassword`/`ResetPassword` помечены `[AllowAnonymous]`.** Буквально ТЗ называет
  анонимными только register/login, но восстановление пароля по определению работает без
  аутентификации (у пользователя нет доступа к аккаунту). Считаю это очевидным намерением
  контракта; `ChangePassword` остаётся защищённым (текущий юзер из claims).
- **Query-параметры forgot/reset/change сделаны `string?` + guard'ы в сервисе.** При non-nullable
  параметрах `[ApiController]` авто-возвращает framework-**ProblemDetails** на пустой/отсутствующий
  параметр — это ломает правило «единый формат `Response<T>`». Nullable + проверки
  `IsNullOrWhiteSpace` (`[NotNullWhen(false)]` — без warning'ов) прогоняют все ошибки через
  `BadRequestException` → exception-middleware → unified `Response<T>`. Проверено рантаймом.
- **Валидация через `ValidateAndThrowAsync`.** `ValidationException` уже маппится middleware в 400
  со списком `errors` — переиспользуем вместо ручного разбора `ValidationResult`.
- **Роль по умолчанию — `DbInitializer.UserRole` ("User").** Одна константа роли на Seed и
  регистрацию, без «магической строки».
