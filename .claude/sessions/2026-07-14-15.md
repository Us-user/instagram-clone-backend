## Решения
- **Группировка в памяти, записи в БД раздельные.** Тянем плоский список получателя (desc по
  времени) с проекцией данных актёра и схлопываем открытые группы через `Dictionary` по ключу —
  O(n), окно 24ч. Пагинация применяется к списку групп (`TotalRecords` = число групп).
- **`Id` группы = самое свежее уведомление**, `IsRead` группы = «все прочитаны». Контрактные
  `mark-as-read?id`/`delete?id` остаются по одной записи (single-id, как в ТЗ).
- **`unread-count` считает записи, а не группы** — привычное поведение бейджа.
- **Enum'ы сериализуются как int** (как `Gender` в базе) — глобальные JSON-опции не трогаю ради
  обратной совместимости; фронт мапит коды.
- **AutoMapper/валидаторы не заводил:** DTO собирается вручную (кастомная группировка, стиль
  `ChatService`), вход — только query-параметры (нет body DTO). Роадмап-пункт закрыт как N/A.
- **`User.IsVerified` добавлен уже сейчас** (не в Phase 19), чтобы не ретрофитить actor-DTO позже.
- **Real-time payload** при создании — одиночная группа из одного актёра (`ReceiveNotification`).
