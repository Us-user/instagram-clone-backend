## Решения
- **Общий `MessageType` вместо отдельного `GroupMessageType`.** §7 требует `System`, §8 — тот же
  набор без него; единый enum `{Text, Image, File, Voice, System}` покрывает оба и Phase 16
  переиспользует его для личных сообщений без дублирования.
- **Непрочитанные на участника через `LastReadAt`.** У групп «прочитано» персонально; храним
  время последнего открытия на `GroupChatMember`, `UnreadCount` = сообщения не свои и не
  служебные с `CreatedAt > LastReadAt`. `get-group-by-id` и `send-message` двигают `LastReadAt`.
- **Reply-самоссылка `ON DELETE SET NULL`.** Удаление процитированного сообщения не должно
  уносить ответы — только обнулять ссылку на цитату (в отличие от cascade у ответов на комменты,
  где нужен каскад). PostgreSQL корректно обрабатывает и это, и каскад группы→сообщения.
- **Авто-передача админства при уходе последнего админа.** Если после `leave` в группе остались
  участники, но не осталось админов — самый давний участник (`JoinedAt`) становится Admin
  (+служебное сообщение). Группа не остаётся без управления; при этом остаётся группой даже с
  1 участником (спец. правило §7).
- **Блок-проверки: create/add-member — да, групповая отправка — нет.** Доступ к отправке в группе
  = членство (общий контекст), попарные блокировки между участниками группу не режут. Но при
  создании отфильтровываются заблокированные (в любую сторону) кандидаты, а `add-member`
  проверяет блок между админом и добавляемым (инвариант #4 соблюдён на входе в группу).
- **Служебные сообщения тоже рассылаются в реал-тайме.** `BroadcastAsync` вызывается после
  add/remove/promote/leave/update — участники видят изменения состава/инфо живьём; на `create`
  рассылка не делается (в UI группы ещё никто, кроме создателя).
- **Тип вложения по расширению.** `send-message` принимает изображения + распространённые файлы
  (25 МБ); `MessageType` = Image для картинок, иначе File. Аватар группы — только изображения
  (дефолт `FileService`). Voice (`MessageType=Voice`, `Duration`/`Waveform`) — в Phase 16.
