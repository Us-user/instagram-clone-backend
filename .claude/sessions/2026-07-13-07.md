## Цель сессии
Полностью закрыть Phase 5: три сервиса + три контроллера (`User`, `UserProfile`,
`FollowingRelationShip`) с дословными путями контракта — поиск пользователей и история
поиска, профиль со счётчиками/isFollowing и изображением, соц-граф подписок. Использовать
`PagedResponse<T>`, `ICurrentUserService` (id из claims), существующие валидаторы/`IFileService`.

## Сделано
- **`Infrastructure/Common/Pagination.cs`** — `Normalize(pageNumber, pageSize)`: страница ≥ 1,
  размер по умолчанию 10, потолок 100. Переиспользуемо для всех пагинированных эндпоинтов.
- **`IUserService` / `UserService`** (10 эндпоинтов):
  - `get-users` — поиск по userName/email (case-insensitive `Contains`) + пагинация →
    `PagedResponse<List<GetUserDto>>`.
  - История текста: `add-search-history` (→ созданный DTO), `get-search-histories`
    (desc по дате), `delete-search-history?id`, `delete-search-histories` (очистка всей).
  - История профилей: `add-user-search-history?UserSearchId` (проверка существования цели,
    запрет на себя, **дедуп «move-to-top»**), `get-user-search-histories`,
    `delete-user-search-history?id`, `delete-user-search-histories`.
  - `delete-user?userId` — **только Admin**; собирает имена файлов (avatar, image профиля,
    все PostImage/Story/Message-файлы юзера) ДО `UserManager.DeleteAsync`, каскады чистят БД,
    затем файлы удаляются с диска.
- **`IUserProfileService` / `UserProfileService`** (7 эндпоинтов):
  - `get-user-profile-by-id?id`, `get-my-profile` — общий приватный `BuildProfileDtoAsync`:
    данные профиля + счётчики постов/подписчиков/подписок + `isFollowing` (для своего
    профиля всегда false).
  - `get-is-follow-user-profile-by-id?followingUserId` → `Response<bool>`.
  - `update-user-profile` — валидатор `UpdateUserProfileDto`, апдейт About/Gender, возврат DTO.
  - `get-post-favorites` — избранные посты текущего юзера, LINQ-проекция `Post → GetPostDto`
    со счётчиками и `isLiked` (`isFavorite = true`), пагинация.
  - `update-user-image-profile` (multipart `imageFile`) — сохранить новый файл, удалить старый,
    вернуть имя; `delete-user-image-profile` — удалить файл + очистить поле.
- **`IFollowingRelationShipService` / `FollowingRelationShipService`** (4 эндпоинта):
  - `get-subscribers?UserId` (кто подписан на UserId → `f.User`),
    `get-subscriptions?UserId` (на кого подписан → `f.FollowingUser`).
  - `add-following-relation-ship?followingUserId` — запрет подписки на себя, проверка
    существования цели, запрет дубля (+ уникальный индекс БД как страховка).
  - `delete-following-relation-ship?followingUserId` — отписка (404 если связи нет).
- **Контроллеры** `UserController`, `UserProfileController`, `FollowingRelationShipController` —
  `[Route("[controller]")]`, паттерн `StatusCode(result.StatusCode, result)`,

## Решения
- **`get-user-profile-by-id?id`: `id` трактуется как `UserId` (string).** Профиль 1:1 с
  пользователем; на руках у клиента именно userId (из `GetUserDto.Id`, из постов, из списков
  подписок), а соседний эндпоинт `get-is-follow-...` тоже принимает userId. Единообразно и удобно.
- **История просмотренных профилей дедуплицируется (move-to-top).** Повторный просмотр удаляет
  прежнюю запись пары (UserId, SearchedUserId) и добавляет свежую — история не растёт дублями,
  порядок отражает последние просмотры (как в поиске Instagram).
- **`delete-user` чистит файлы с диска до удаления сущности.** Каскады EF убирают строки, но не
  файлы; имена собираются заранее (avatar, image профиля, PostImage/Story/Message юзера), удаление
  с диска — после успешного `DeleteAsync`. Выполняет правило ТЗ №6 о чистке файлов.
- **Проекция `Post → GetPostDto` написана LINQ-запросом (а не AutoMapper).** Нужны вычисляемые
  поля (`isLiked` для текущего юзера, счётчики) — прямой `Select` переводится в SQL и не тянет
  лишние навигации. В Phase 6 эту проекцию переиспользует `PostService`.
- **Параметры-строки query сделаны `string?` + guard'ы в сервисе.** Как в Phase 4: non-nullable
  параметр заставил бы `[ApiController]` вернуть framework-ProblemDetails вместо unified
  `Response<T>`. Nullable + `BadRequestException` прогоняет ошибку через exception-middleware.
- **`AsNoTracking()` на всех read-запросах.** Списки/профили только читаются — трекинг не нужен.
